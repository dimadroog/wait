# TRAIN_ANALYSIS — как читать лог обучения

> Операторский разбор консоли `train_ppo` / `train_local.sh`.  
> Термины: [GLOSSARY.md](GLOSSARY.md) · CLI: [SCRIPTS.md](SCRIPTS.md) · замеры скорости: [MEASUREMENTS.md](MEASUREMENTS.md)

**Зачем документ:** по цифрам в консоли вовремя понять, есть ли прогресс политики или пора остановить прогон и пересмотреть настройки — **без обязательного визуального inference**.

Лог печатается после каждого [rollout](GLOSSARY.md#rollout) (сбор опыта + обновление [PPO](GLOSSARY.md#ppo)). Обычно видны два потока:

| Поток | Откуда | Когда |
| ----- | ------ | ----- |
| Таблица Stable-Baselines3 | `verbose=1` + поля прогресса из callback | почти всегда при обучении |
| Строка `rollout_metrics:` | `RolloutMetricsCallback` | только с флагом `--rollout-metrics` (+ JSONL в `tmp/bench/…`) |

Кумулятивное `fps` в таблице **не** то же самое, что `rate` за один rollout. Деградацию скорости смотрите по `rate` / `wall` ([MEASUREMENTS § R6](MEASUREMENTS.md#r6-dual-trainmeasure)).

---

## Практическое правило остановки (политика)

Смотрите блок `train/` и «залипание» эпизодных метрик, а не только «награда высокая».

Остановите прогон и пересмотрите политику / BC / награду, если **несколько выводов подряд** (ориентир: **≥ 3**, надёжнее **≥ 10**) одновременно:

1. `approx_kl` ≈ 0 и `clip_fraction` = 0;
2. `|entropy_loss|` очень мала (порядка **0.01** и меньше; у «живой» политики часто доли единицы, например около −0.4);
3. `ep_rew_mean` и `ep_len_mean` не меняются (бит-в-бит одни и те же значения).

Высокая награда при этих трёх пунктах **не** отменяет стоп: награда может начисляться за стояние или ложные срабатывания контрольных точек.

## Параметры таблицы Stable-Baselines3

Колонки ниже — для чтения **динамики между соседними выводами** в консоль (каждый вывод ≈ один завершённый rollout).

| Название параметра | Динамика | Критичность для остановки обучения | Описание параметра |
| --- | --- | --- | --- |
| `ep_len_mean` | При осмысленном обучении обычно **растёт** или стабилизируется на разумной длине попытки. Резкие скачки возможны при смене режима смерти. Не должна быть «вечной константой», если вы ждёте нового поведения. | **Стоп / пересмотр:** одно и то же значение **много выводов подряд** (ориентир ≥ 10) вместе с мёртвым блоком `train/` (см. правило выше). Само по себе короткое значение зависит от режима смерти в конфиге игры и не всегда баг. | Средняя длина законченного игрового эпизода в [шагах среды](GLOSSARY.md#env-step) за недавнее окно. Это не кадры NES и не «насколько далеко пробежал герой». |
| `ep_rew_mean` | При прогрессе **растёт** или хотя бы **медленно плывёт**. «Высоко и идеально ровно» без изменения поведения — подозрительно. | **Стоп / пересмотр:** значение **не меняется** много выводов подряд при живом `total_timesteps`. Не останавливайтесь только потому, что число «маленькое» или «большое» — смотрите **изменение** и связку с `entropy_loss` / `approx_kl`. | Средняя суммарная [награда](GLOSSARY.md#rl) за эпизод. Красивая цифра не доказывает, что агент нажимает осмысленные кнопки. |
| `fps` | На длинной сессии часто **медленно падает** (нагрузка машины). Для прогресса политики не главный индикатор. | **Стоп по железу / скорости:** сильное падение относительно начала сессии (например в разы) или уход к ~1 при том, что раньше было ~5–8 — проверьте RAM/процессы; это не про «умность» модели. | **Не** частота кадров видео. Кумулятив: все шаги среды с начала `learn()`, делённые на [wall-clock](GLOSSARY.md#wall-clock). Сглаживает просадки отдельного rollout. |
| `iterations` | **Растёт** на +1 каждый вывод таблицы. | **Стоп:** не растёт при «живой» консоли долго — процесс завис (см. также `--learn-stall-timeout`). | Сколько rollout уже завершено с начала текущего `learn()`. |
| `time_elapsed` | **Растёт** (секунды wall-clock с старта `learn()`). | Сама по себе не стоп-критерий политики. Резкий скачок шага между строками = этот rollout (+ PPO) шёл очень долго. | Сколько реального времени прошло с начала обучения по часам компьютера. |
| `total_timesteps` | **Растёт** примерно на `n_envs × n_steps` за rollout (например 6×128 = 768). | **Стоп:** не растёт N минут — зависание сбора опыта (есть watchdog `--learn-stall-timeout`). | Сколько шагов среды уже сделано суммарно (`num_timesteps` модели). |
| `progress_pct` | **Растёт** к 100 за **текущий** бюджет `learn`. | Не стоп по качеству политики. 100% = исчерпан бюджет этого прогона, не «миссия пройдена». | Доля `100 × (total_timesteps − start) / (target − start)`. При новом PPO / `--model-in` счётчик сбрасывается — шкала 0→100% по `remaining`; при `--resume` — от уже накопленных шагов к абсолютной цели. Отключается `--no-progress-pct`. |
| `target_timesteps` | В консоли — знаменатель шкалы прогресса текущей сессии (при сбросе счётчика = `remaining`). В `.train.json` по-прежнему абсолютная цель CLI. | Не стоп-критерий. Если абсолютная цель CLI меньше уже накопленных шагов модели — обучение сразу выйдет («target already reached»). | Не путать консольный `time/target_timesteps` (окно прогресса) с полем sidecar. |
| `approx_kl` | Должна быть **маленькой, но не вечным нулём**: политика чуть меняется каждый rollout. Рост слишком большой — обновления политики резкие (осторожно с learning rate). | **Стоп / пересмотр политики:** **≈ 0.0** на протяжении многих выводов (≥ 3–10) вместе с `clip_fraction = 0` — обновлений поведения почти нет. | Оценка «насколько новый выбор кнопок ушёл от старого» за обновление PPO. Ноль долго = политика застыла. |
| `clip_fraction` | Обычно **то больше нуля, то около нуля** (доля слишком резких обновлений, которые PPO обрезал). | **Стоп / пересмотр:** **всегда 0** много выводов подряд при `approx_kl ≈ 0` — клипать нечего, политика не сдвигается. | Доля шагов обновления, где сработало ограничение PPO на величину изменения политики. |
| `clip_range` | Обычно **константа** (часто 0.2) — гиперпараметр, не метрика прогресса. | Не стоп-критерий сам по себе. | Насколько PPO разрешает политике «прыгнуть» за одно обновление. |
| `entropy_loss` | При живом исследовании модуль обычно **не около нуля** (в логе часто отрицательное число вроде −0.4). По мере обучения может **умеренно приближаться к нулю**, но не обязана мгновенно схлопнуться. | **Стоп / пересмотр:** модуль **≈ 0** (например −0.001) и держится — политика почти перестала пробовать разные кнопки (типичный фон для «всегда стоять»). Особенно тревожно сразу после [behavioral cloning](GLOSSARY.md#behavioral-cloning). | Число из лога, связанное с «случайностью» выбора кнопки: насколько политика ещё пробует разные действия, а не один и тот же ответ. Около нуля ≈ почти наверняка одно и то же действие (часто «ничего не жать»). |
| `explained_variance` | Желательно **растёт** к 1 (критик лучше предсказывает возврат). | **Не** стоп сама по себе. Рост при мёртвой политике (`approx_kl≈0`, энтропия≈0) значит: учится оценка награды, а не кнопки — всё равно нужен пересмотр. | Насколько хорошо сеть-критик объясняет фактические суммы наград. Не говорит, бежит ли герой. |
| `learning_rate` | Обычно **константа** (или по расписанию, если задано). | Не стоп-критерий политики. | Шаг градиентного обновления весов. |
| `loss` | Часто **медленно падает** или колеблется; ориентир вторичный. | Не единственный стоп. Падение `loss` при нулевом `approx_kl` не доказывает прогресс в игре. | Сводный loss обучения за обновление (смесь компонент PPO). |
| `n_updates` | **Растёт** по мере обновлений PPO. | Не стоп-критерий, если просто растёт. | Сколько раз уже обновили веса оптимизатором. |
| `policy_gradient_loss` | Должна **шевелиться** (ненулевой шум порядка, заметного глазу). | **Стоп / пересмотр:** величины порядка **1e-7** и меньше долго — градиент политики практически мёртв. | Часть ошибки, которая двигает «какую кнопку жать». |
| `value_loss` | Часто **падает** или стабилизируется по мере обучения критика. | Сама по себе редко стоп. Падение при noop-политике не спасает поведение. | Ошибка критика: насколько плохо предсказана ценность состояния. |

## Параметры строки `rollout_metrics:` (опционально)

Включается `--rollout-metrics`. Нужна в первую очередь для **скорости машины**, не для «умеет ли агент играть».

| Название параметра | Динамика | Критичность для остановки обучения | Описание параметра |
| --- | --- | --- | --- |
| `wall` (`wall_rollout_s`) | Желательно **относительно стабильна** от rollout к rollout после прогрева. | **Стоп / расследование железа:** устойчивый рост в **2× и больше** к ранним rollout (см. `parse_train_rollouts.py`, флаг degraded) — сессия деградирует по времени. | Сколько wall-секунд занял **только этот** сбор опыта до конца rollout. |
| `steps` (`delta_timesteps`) | Обычно **константа** ≈ `n_envs × n_steps`. | **Стоп:** систематически 0 или сильно меньше ожидаемого — сбой сбора. | Сколько шагов среды набрано в этом rollout. |
| `rate` (`env_steps_per_s`) | Желательно держаться около ранней базы сессии (порядок единиц–десятков на вашем железе). | **Стоп / расследование:** падение в разы относительно начала длинного прогона — сначала железо/RAM, не «докачать политику». | Скорость **этого** rollout: `steps / wall`. Удобнее `fps` из таблицы для ловли просадки. |
| `avail_ram_mb` | Не должна уходить в критически низкие значения на вашей машине. | **Стоп:** свободная RAM тает к сотням МБ и ниже при росте `wall` — риск swap и обвала скорости. | Свободная физическая память Windows (МБ). В JSONL есть ещё total и процент загрузки. |

Сводка JSONL: [`parse_train_rollouts.py`](SCRIPTS.md#parse_train_rolloutspy).

## Пример вывода

`n_envs=6`, `n_steps=128`, включён `--rollout-metrics`:

```
rollout_metrics: #2 wall=95.1s steps=768 rate=8.079 avail_ram_mb=9262.2
---------------------------------
| rollout/           |          |
|    ep_len_mean     | 2        |
|    ep_rew_mean     | -40      |
| time/              |          |
|    fps             | 4        |
|    iterations      | 2        |
|    time_elapsed    | 412      |
|    total_timesteps | 1536     |
|    progress_pct    | 1.5      |
|    target_timesteps| 100000   |
---------------------------------
```

В полном логе PPO ниже часто есть блок `train/` с `approx_kl`, `entropy_loss`, `clip_fraction` и др. — для стопа политики читайте **его** в первую очередь.
